## 1. 项目定位与 DDD 战略设计

### 一句话定义

> FinHub 是一套基于 **DDD 领域模型** + Java + MyBatis-Plus + AI 的跨平台资金流水聚合引擎。以**资金流水上下文**为核心域，以**查询分析**与 **AI 辅助**为支撑域，通过防腐层隔离外部 CSV 格式与 LLM 技术细节，支持 Docker 一键部署。

### 核心能力（三句话）

-   **导入** ：通过防腐层（ACL）将支付宝/微信/银行 CSV 转换为统一领域语言，聚合根（Transaction）守护排重与金额不变量。
-   **排重** ：三重防重机制（MySQL 唯一索引 + 应用层预检 + 模糊指纹兜底）作为**领域服务**（DeduplicationService）实现，聚合根内不直接访问仓库，保持纯粹。
-   **查询** ：规则路由优先（70% 常见问题走 SQL 读模型），复杂语义经 AI 辅助上下文翻译后，通过 AST 白名单校验，以只读方式穿透防腐层查询核心域。

### 核心设计哲学

> **领域模型优先，技术实现兜底；AI 只做翻译，不做计算；限界上下文内保持统一语言，上下文间通过防腐层协作。**

-   金额精度、排重规则、资金流向 → **100% 封装在值对象与领域服务中**
-   CSV 解析、LLM 调用、缓存实现 → **全部隔离在防腐层与基础设施层**
-   自然语言转 SQL、异常项语义解释 → **支撑域能力，经 ACL 进入核心域**
-   Prompt、Golden Set、Few-shot → **数据治理上下文资产，版本化管理**

### DDD 战略设计：子域与限界上下文划分

| 子域类型 | 限界上下文 | 职责 | 统一语言关键词 |
| :--- | :--- | :--- | :--- |
| **核心域** | **资金流水上下文（Fund Flow Context）** | 交易聚合根生命周期、排重规则、金额不变量、导入批次 | Transaction、Money、Fingerprint、ImportBatch、Deduplication |
| **支撑域** | **查询分析上下文（Query Analysis Context）** | 规则路由、聚合统计、关键词检索、读模型优化 | QueryRouter、AggregationView、KeywordSearch |
| **支撑域** | **AI 辅助上下文（AI Assistance Context）** | 自然语言转 SQL、语义解释、Prompt 管理 | NLTranslator、ValidatedSql、PromptRegistry |
| **通用域** | **数据治理上下文（Data Governance Context）** | Prompt/Golden Set 版本化、DI 流水线、数据血缘 | DvcPipeline、GoldenSet、EvalCase |
| **通用域** | **基础设施与部署上下文（Infra Context）** | 容器编排、健康检查、安全基线、日志脱敏 | DockerCompose、EncryptedValue、AuditLog |

## 2. 技术栈与架构原则

### 技术栈清单

-   Java 17 + Spring Boot 3.2 + Maven
-   **MyBatis-Plus 3.5.x（基础设施层实现 Repository，领域层仅依赖接口）**
-   **MySQL 8.0（聚合根持久化，Flyway 管理聚合 schema 演进）**
-   Flyway（手写 SQL 迁移，禁止 MP 自动 DDL，保证 schema 与领域模型同步）
-   Caffeine（基础设施层缓存，支撑领域服务排重预检）
-   Spring AI（支撑域：AI 辅助上下文的技术实现）
-   JSqlParser（支撑域：SQL AST 解析，作为防腐层校验器）
-   DVC / Git LFS（数据治理上下文：Prompt/Golden Set 版本化）
-   **Docker + Docker Compose（基础设施上下文：环境一致性）**
-   Springdoc OpenAPI（接口层文档）
-   JUnit 5 + AssertJ（领域层 100% 覆盖优先）

### 八条不可违背的架构原则

1.  **聚合根不变量绝对化** ：Transaction 聚合根在构造和修改时必须保证金额有效（Money 非空、精度正确）、指纹生成规则一致、敏感字段加密。任何违反不变量的状态不允许存在。
2.  **金额精度值对象化** ：Money 为不可变值对象，内部强制 `BigDecimal` + `DECIMAL(18,2)`，提供行为方法（add/compareTo/abs），禁止在应用层直接操作原始 BigDecimal。
3.  **排重规则领域服务化** ：跨聚合/跨记录的排重判断不放在聚合根内（避免仓库依赖注入污染实体），统一由 DeduplicationService 领域服务执行，但其规则（什么是重复）属于领域知识。
4.  **AI 输出必经防腐层校验** ：AI 辅助上下文返回的 SQL 必须通过 AST 白名单解析（防腐层校验），不通过则丢弃；禁止穿透至核心域。
5.  **扩展点接口化（防腐层）** ：DataSourceAdapter、NLTranslator、KnowledgeBaseAdapter 全部定义为防腐层接口，核心域不依赖任何外部技术细节。
6.  **敏感数据值对象加密化** ：Counterparty、Remark 等敏感字段封装为 `EncryptedString` 值对象，入库即密文，密钥通过容器环境变量注入，禁止硬编码。
7.  **数据资产版本化** ：Prompt、Golden Set、Few-shot 纳入数据治理上下文，通过 DVC 管理，禁止硬编码在代码中。
8.  **部署容器化** ：所有上下文依赖通过 Docker Compose 编排，数据库数据卷持久化，禁止本地直连远程库。

## 3. 系统全景架构：限界上下文与分层映射

### DDD 分层架构映射（面试必讲）

| DDD 分层          | 职责                                   | 核心组件（按上下文归属）                                     | 技术实现                                  |
| :---------------- | :------------------------------------- | :----------------------------------------------------------- | :---------------------------------------- |
| **用户/接口层**   | REST API、DTO 转换、文件上传、基础认证 | Controller、DTO、SecurityFilter                              | Spring MVC、Spring Security（Basic Auth） |
| **应用层**        | 业务用例编排、事务边界、领域事件发布   | IngestionAppService、QueryAppService、EventPublisher         | Spring @Transactional、ApplicationEvent   |
| **领域层**        | 业务规则、不变量、值对象行为、领域服务 | **资金流水上下文**：Transaction（聚合根）、Money、Fingerprint、DeduplicationService、FingerprintGenerator | 纯 Java，无 Spring 注解（除领域事件可选） |
| **防腐层（ACL）** | 隔离外部技术细节，转换为领域语言       | DataSourceAdapter、NLTranslatorAdapter、RawTransactionFactory | 接口定义在领域层，实现可依赖 Spring       |
| **基础设施层**    | 持久化、缓存、外部技术实现             | TransactionRepositoryImpl（MP）、CacheImpl（Caffeine）、SpringAIClient | MyBatis-Plus、Caffeine、Spring AI         |
| **🆕 部署层**      | 容器编排、健康检查、数据卷             | Dockerfile、docker-compose.yml、Actuator                     | Docker、Spring Boot Actuator              |

### 关键战术设计：聚合与值对象

#### 资金流水上下文 — Transaction 聚合

-   **聚合根：Transaction（实体）**
    -   唯一标识：数据库自增 ID（领域标识） + external_id（业务标识）
    -   生命周期：创建 → 排重校验 → 持久化 → 发布领域事件
    -   不变量：amount 必须为有效 Money；direction 不能为 null；若 external_id 缺失则 fingerprint 必须生成。

-   **值对象：Money**
    -   不可变，构造时强制精度（保留 2 位小数，四舍五入）。
    -   行为：提供 `add()`、`subtract()`、`abs()`、`compareTo()`，返回新实例。
    -   面试点：值对象无 ID，可自由替换；`new Money("100.00")` 与 `new Money("100.0")` 在业务上等价。

-   **值对象：Fingerprint**
    -   封装排重指纹（SHA256 结构化哈希 + 盐值）。
    -   生成逻辑复杂（涉及时间截断、空值占位、盐值混合），不放在构造函数，由 **FingerprintGenerator 领域服务** 工厂化生成。
    -   面试点：值对象内聚业务规则，但复杂生成过程委托给领域服务，避免实体臃肿。

-   **值对象：EncryptedString**
    -   封装敏感字段（counterparty、remark）的密文形态。
    -   构造时即加密（或工厂方法加密），提供 `decrypt()` 方法需传入密钥。
    -   面试点：安全是业务属性，不是技术细节，因此加密逻辑封装在值对象中。

-   **领域服务：DeduplicationService**
    -   职责：执行跨记录排重判断（查询仓库、比对指纹）。
    -   为什么不是聚合根方法？因为排重需要读取已有聚合，聚合根不应持有 Repository 依赖。
    -   三重防重：
        1.  优先比对 external_id（业务唯一标识）
        2.  external_id 缺失时，比对 fingerprint（结构化模糊指纹）
        3.  应用层 Caffeine 预检 + DB 唯一约束兜底（基础设施配合）

-   **领域服务：FingerprintGenerator**
    -   职责：根据金额、时间（截断到分钟）、对方户名（标准化）、备注（空值占位）生成带盐指纹。
    -   处理边界：同一分钟多笔相同金额消费（如自动售货机连刷）的误判风险。

-   **领域事件**
    -   `TransactionImportedEvent`：导入成功后发布，可用于后续统计更新（MVP 可空消费，但预留扩展）。
    -   `DuplicateDetectedEvent`：发现重复时发布，用于审计日志。

#### 查询分析上下文 — 读模型（CQRS 预留）

-   MVP 阶段不强制物理分离数据库，但逻辑上区分：
    -   **写模型**：Transaction 聚合（资金流水上下文），负责导入、排重、金额不变量。
    -   **读模型**：AggregationView、KeywordSearchResult（查询分析上下文），直接通过 MP Mapper 查询，允许为读优化而轻微反规范化。
-   面试点："我预留了 CQRS 演进路径，MVP 阶段共享数据库但逻辑分离，未来可拆分为独立读库。"

#### AI 辅助上下文 — 防腐层设计

-   **防腐层接口：NLTranslator**
    -   输入：NaturalLanguageQuery（值对象，封装原始提问 + 用户上下文）
    -   输出：ValidatedSql（值对象，封装 SQL + 校验状态）
    -   核心域不依赖此接口，由应用层 QueryAppService 按需调用。

-   **防腐层实现：SpringAiNLTranslator**
    -   依赖 Spring AI 调用 OpenAI/Claude，从 PromptRegistry（数据治理上下文）加载模板。
    -   输出 SQL 经 SqlAstValidator 校验（仅允许 SELECT + 白名单表 + 无子查询），失败则返回降级提示。

-   **面试点**：AI 上下文通过 ACL 与核心域交互，核心域的 Transaction 聚合根本不知道"自然语言"是什么，保证核心域纯粹。

### Phase 2：领域层核心（Day 2）

-   **Transaction 聚合根**：
    -   字段：id、external_id、money（Money VO）、direction（Enum）、time、counterparty（EncryptedString VO）、remark（EncryptedString VO）、fingerprint（Fingerprint VO）、sourceSystem（Enum）。
    -   工厂方法：`Transaction.createFrom(RawRecord record, Fingerprint fingerprint)`，确保构造时即通过排重预检。
    -   不变量校验：amount 不能为 null；direction 与 amount 符号一致性（收入为正、支出为负）。
-   **Fingerprint 值对象**：
    -   封装 hash 与 salt，提供 `matches()` 方法。
    -   不暴露生成算法，由 FingerprintGenerator 工厂创建。
-   **EncryptedString 值对象**：
    -   封装密文，构造时传入明文 + 密钥，内部完成 AES 加密。
    -   提供 `decrypt(key)` 返回明文，用于查询时展示（但日志中禁止打印）。
-   **领域事件**：定义 TransactionImportedEvent、DuplicateDetectedEvent（MVP 可空消费，但必须在代码中预留发布点）。
-   **Flyway SQL**：创建 transactions 表。
    -   external_id 唯一约束；fingerprint 唯一约束；amount DECIMAL(18,2)；counterparty/remark 为密文字段（TEXT）。
    -   预留 cost_center、project_code 等企业扩展字段。

**面试价值**：展示聚合根不变量设计 + 值对象不可变性 + 安全内聚到领域层。

### Phase 3：防腐层 + 应用层编排（Day 3）

-   **防腐层接口：DataSourceAdapter**
    -   定义在资金流水上下文的 `acl` 包，或领域层的 `factory` 包。
    -   方法：`List<RawRecord> adapt(InputStream csv)`。
    -   面试点：核心域不知道"CSV"是什么，只认识 RawRecord（领域概念）。
-   **实现：AlipayCSVAdapter**
    -   仅支持 2024 新版支付宝格式。
    -   异常行（如退款记录格式不一致）跳过并记录告警，不阻断整个文件。
    -   编码处理：自动识别 UTF-8 / GBK。
-   **应用层：IngestionAppService**
    -   编排用例：接收文件 → 调用 Adapter（防腐层）→ 生成 Fingerprint（领域服务）→ 调用 DeduplicationService（领域服务）→ 创建 Transaction（聚合根）→ 调用 Repository（基础设施）→ 发布领域事件。
    -   事务边界在此层（@Transactional），包裹批量插入。
-   **领域服务：DeduplicationService**
    -   三重防重逻辑：
        1.  查 external_id（业务唯一标识）
        2.  查 fingerprint（结构化模糊指纹，含盐值）
        3.  Caffeine 预检（基础设施配合）+ DB 唯一约束兜底
    -   返回：去重后的 Transaction 列表 + 被拦截的重复列表（发布 DuplicateDetectedEvent）。
-   **领域服务：FingerprintGenerator**
    -   结构化指纹算法：金额截断精度 + 时间截断到分钟 + 对方户名标准化（去空格/特殊字符）+ 备注空值占位（`__EMPTY__`）+ 盐值。
    -   处理边界：同一分钟多笔相同金额消费（如自动售货机连刷）的误判风险。

**面试价值**：展示防腐层设计 + 应用层与领域层职责分离 + 领域服务处理跨聚合逻辑。

### Phase 4：基础设施层实现（Day 4）

-   **TransactionRepositoryImpl**：实现领域层定义的 `TransactionRepository` 接口。
    -   内部组合 MyBatis-Plus BaseMapper，但对外隐藏 MP 细节。
    -   复杂聚合统计（如按月汇总）手写 XML，避免 MP 生成低效 SQL。
    -   批量插入：开启 MySQL `rewriteBatchedStatements=true`，或手写 `INSERT INTO ... VALUES (...), (...)`。
-   **CaffeineCacheImpl**：实现缓存预检，但明确：
    -   仅作为降低 DB 查询压力的加速器，不充当持久化排重权威。
    -   配置 `maximumSize(10_000)` + `expireAfterWrite(30min)`，防止容器重启后缓存丢失导致重复导入。
-   **单元测试**：领域层（Money、Fingerprint、DeduplicationService）100% 覆盖；应用层编排逻辑 > 80%。

**面试价值**：展示 Repository 模式（领域层定义接口，基础设施实现）+ MP 批量优化 + 缓存边界意识。

### Phase 5：查询分析上下文（Day 5）

-   **应用层：QueryAppService**
    -   编排查询用例：接收查询 → 调用 QueryRouter → 返回结果。
-   **QueryRouter（责任链）**：
    -   正则匹配 → 规则路由（MP LambdaQueryWrapper / 手写 XML）→ AI 辅助上下文（支撑域）兜底。
    -   面试点：简单问题快速响应，复杂问题才走 AI，控制成本。
-   **读模型**：AggregationView（DTO），允许为查询性能反规范化，与 Transaction 聚合根解耦。
-   **关键词检索**：LIKE 查询（预留 Elasticsearch 演进路径）。
-   **性能要求**：聚合查询（10 万行数据）< 100ms，需通过 `EXPLAIN` 验证索引。

**面试价值**：展示 CQRS 预留 + 责任链模式 + SQL 优化意识。

### Phase 6：AI 辅助上下文（Day 6）

-   **防腐层接口：NLTranslator**
    -   输入：NaturalLanguageQuery（值对象）
    -   输出：ValidatedSql（值对象，含 SQL + 校验状态 + 失败原因）
-   **实现：SpringAiNLTranslator**
    -   从 PromptRegistry（数据治理上下文）加载模板，禁止硬编码 Prompt。
    -   记录 Token 消耗与响应时间（用于 BENCHMARK.md）。
-   **SqlAstValidator**：
    -   使用 JSqlParser 解析 AST，白名单校验：仅允许 SELECT + 指定表 + 无子查询 + 无 DML。
    -   失败时返回降级提示，不穿透至核心域。
-   **集成**：QueryAppService 在责任链末端调用 NLTranslator，结果经校验后作为只读 SQL 执行。

**面试价值**：展示 ACL 保护核心域 + AI 成本意识 + 安全红线设计。

### Phase 7：数据治理 + CI/DI（Day 7）

-   **数据治理上下文**：
    -   DVC 配置：管理 `prompts/`、`golden-set/` 目录。
    -   Golden Set：20 条真实查询 + 错误模式记录，转化为 Eval 用例。
    -   每次 Prompt 变更必须有 Diff 记录。
-   **CI 流水线**：GitHub Actions，编译 + 单元测试（领域层 100% 覆盖）+ 打包镜像。
-   **DI 流水线**：DVC Pipeline，验证 Golden Set 通过率。

**面试价值**：展示数据资产与代码同等重要 + AI 工程化素养。

### Phase 8：安全加固 + 容器化（Day 8）

-   **安全基线落地**：
    -   验证 EncryptedString 在数据库中为密文。
    -   日志脱敏：配置 SLF4J 过滤器，金额 > 1000 或敏感字段自动打码。
    -   基础认证：Spring Security Basic Auth，用户名/密码通过环境变量注入。
    -   临时文件清理：容器启动时清空 `/tmp/finhub/upload/`。
-   **Docker 调优**：
    -   docker-compose.yml 增加健康检查（Actuator `/actuator/health`）。
    -   MySQL 数据卷挂载到宿主机，防止容器误删导致数据丢失。
    -   增加备份服务注释（如 `offen/docker-volume-backup`），展示容灾意识。
-   **镜像优化**：多阶段构建，目标体积 < 100MB。

**面试价值**：展示安全底线 + 容器可观测性 + 数据不可再生意识。

### Phase 9：整合测试 + 修坑（Day 9）

-   **端到端测试**：CSV 上传 → 解析 → 排重 → 入库 → 查询 → 聚合，全流程验证。
-   **修坑**：处理 Docker 网络、MySQL 时区、CSV 编码、MP 批量插入性能等意外问题。
-   **BENCHMARK.md**：记录性能基线与数据质量。
    -   1000 行 CSV：解析 + 排重 + 入库 < 500ms，内存 < 150MB。
    -   10000 行 CSV：解析 + 排重 + 入库 < 2s，内存 < 200MB。
    -   导入成功率 > 99.5%，异常行比例 ~0.3%。
    -   规则路由拦截率 > 70%，AI 查询响应时间 ~2s。
    -   容器镜像体积 ~85MB，启动时间 ~3s。
-   **ADR 补全**：
    -   ADR-000：DDD 战略设计决策

    -   ADR-002：防腐层预留（ToolExecutor + KnowledgeBaseAdapter）

    -   ADR-004：DVC 数据资产版本化
    -   ADR-005：MVP 安全基线（字段加密 + 日志脱敏）
    -   ADR-006：Repository 模式 + MP 实现分离
-   **演示视频**：录制 3 分钟演示（导入 → 排重 → 聚合查询 → AI 查询）。

## 6. 面试价值点清单（含 DDD 决策与失败案例）

### 必讲六个架构决策（ADR）

| ADR         | 决策内容                            | 面试话术                                                     |
| :---------- | :---------------------------------- | :----------------------------------------------------------- |
| **ADR-000** | **为什么采用 DDD 而非贫血 MVC**     | "资金流水有严格的不变量（金额精度、排重规则），贫血模型会导致规则散落在 Service 中易被绕过。DDD 聚合根确保状态与行为内聚，Money 值对象让金额精度不可能被业务代码忽略。" |
| **ADR-001** | **Money 值对象 + BigDecimal**       | "账务系统绝不允许精度丢失。Money 是不可变值对象，构造时即强制精度，提供 add/compareTo 行为，业务代码无法直接操作裸 BigDecimal。" |
| **ADR-002** | **防腐层隔离 CSV 与 AI**            | "核心域的 Transaction 聚合根根本不知道'CSV'或'OpenAI'是什么。外部格式变化（支付宝改字段）只影响 Adapter 实现，核心域稳定。" |
| **ADR-003** | **AI 只做翻译 + AST 校验**          | "AI 辅助上下文通过 ACL 与核心域交互，输出 SQL 必须经 AST 白名单解析。我画了明确红线：AI 不能直接接触业务数据。" |
| **ADR-004** | **Prompt/Golden Set 用 DVC 版本化** | "AI 应用中数据和代码同等重要，必须像管理代码一样管理数据资产。" |
| **ADR-005** | **MVP 字段加密 + 日志脱敏**         | "个人资金数据不可再生，即使 MVP 也做 AES 加密。对方户名和备注在数据库中是密文，日志开启脱敏，这是安全底线。" |

### 必讲 DDD 战术设计亮点

| 设计点                             | 实现                     | 面试话术                                                     |
| :--------------------------------- | :----------------------- | :----------------------------------------------------------- |
| **聚合根：Transaction**            | 封装导入生命周期与不变量 | "Transaction 是聚合根，构造时必须通过 Money 值对象校验和 Fingerprint 生成。外部无法创建一个'金额非法'或'未排重'的 Transaction。" |
| **值对象：Money**                  | 不可变、行为丰富         | "Money 是值对象，无 ID，100.00 与 100.0 业务等价。它提供 add/subtract 方法，返回新实例，保证不可变性。" |
| **值对象：Fingerprint**            | 复杂生成委托领域服务     | "指纹生成涉及时间截断、空值占位、盐值混合，逻辑复杂。我把它放在 FingerprintGenerator 领域服务中，避免聚合根臃肿。" |
| **领域服务：DeduplicationService** | 跨记录排重               | "排重需要读取已有聚合，聚合根不应持有 Repository。我放在领域服务中，既保持实体纯粹，又内聚业务规则。" |
| **防腐层：DataSourceAdapter**      | 隔离 CSV 技术细节        | "核心域只认识 RawRecord，不认识 CSV。支付宝改字段格式，只需改 Adapter，Transaction 聚合根完全无感。" |
| **领域事件**                       | TransactionImportedEvent | "导入成功后发布事件，MVP 空消费，但未来可扩展为'自动更新预算分类'或'发送通知'，核心域与后续处理解耦。" |

### 必讲三个设计模式（DDD 视角）

| 模式           | 用在哪儿                                    | 面试话术                                                     |
| :------------- | :------------------------------------------ | :----------------------------------------------------------- |
| **策略模式**   | DataSourceAdapter（防腐层）                 | "新增渠道只需加一个 Adapter 实现，核心域通过统一接口接入，符合开闭原则。这是防腐层的经典应用。" |
| **责任链模式** | QueryRouter（应用层）                       | "简单问题快速响应，复杂问题才走 AI，控制成本。每条规则是独立处理者，符合单一职责。" |
| **工厂模式**   | FingerprintGenerator + Transaction 工厂方法 | "复杂对象（Fingerprint、Transaction）的创建委托给工厂/领域服务，保证创建即合法，避免半成品对象在系统中流转。" |

### 必讲五个非功能性设计

| 设计           | 实现方式                                               | 面试话术                                                     |
| :------------- | :----------------------------------------------------- | :----------------------------------------------------------- |
| **幂等性**     | external_id 唯一索引 + Fingerprint 值对象 + 应用层预检 | "用户重复上传同一份文件，或 CSV 字段错位导致 ID 缺失，都不会产生脏数据。Fingerprint 值对象保证相同业务数据生成相同指纹。" |
| **降级策略**   | AI 超时/报错/校验失败 → 返回默认提示                   | "AI 辅助上下文故障时，QueryRouter 回退到规则查询，系统依然可用，只是功能降级。" |
| **可观测性**   | SLF4J 日志（脱敏）+ 每步耗时记录 + Actuator 健康检查   | "生产环境出问题，能快速定位是解析层、领域层还是 AI 层。日志脱敏保证安全。" |
| **部署一致性** | Docker Compose 编排                                    | "开发、测试、生产环境完全一致，杜绝'在我机器上能跑'。"       |
| **数据安全**   | EncryptedString 值对象 + 日志脱敏 + Basic Auth         | "敏感信息封装为值对象，入库即密文。这是业务属性，不是后期打补丁。" |

### 必讲两个失败/妥协案例（差异化关键）

| 案例                                   | 初始方案                                               | 失败原因                                                     | 最终妥协方案                                                 | 面试话术                                                     |
| :------------------------------------- | :----------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **贫血模型导致规则散落**               | 传统 MVC：Transaction 为 POJO，业务逻辑在 Service      | 金额校验、排重规则散落在多个 Service 中，新人开发时遗漏校验，导致脏数据入库 | 改为 DDD：Money/Fingerprint 值对象 + Transaction 聚合根不变量 + DeduplicationService 领域服务 | "我曾用贫血模型做资金系统，结果金额校验散落在三个 Service 里，被绕过导致脏数据。改为 DDD 聚合根后，非法状态根本无法创建，这是我从失败中学会的'防御式编程'。" |
| **MP 滥用导致慢查询 + 误用 saveBatch** | 所有查询用 LambdaQueryWrapper，批量插入用 MP saveBatch | 复杂聚合 SQL 无法利用联合索引；saveBatch 底层是逐条 INSERT，1 万行 CSV 耗时 30 秒 | 复杂统计手写 XML + 索引优化；批量插入开启 `rewriteBatchedStatements=true` 或手写批量 SQL | "我曾过度依赖 MP 的便捷性，结果发现它生成的聚合 SQL 全表扫描，saveBatch 也不是真批量。这让我认识到：ORM 是效率工具，但不能替代 SQL 优化思维，Repository 接口可以隐藏这些技术细节。" |

### 核心改进点总结（相比原版）

| 维度 | 原版 | DDD 版 |
|------|------|--------|
| **架构范式** | 传统分层（Controller-Service-Mapper） | **DDD 分层**（接口-应用-领域-基础设施-防腐层） |
| **业务边界** | 按技术分层 | **按限界上下文划分**（资金流水、查询分析、AI 辅助、数据治理） |

| **CSV/AI 隔离** | 接口 + 实现 | **防腐层（ACL）**，核心域不认识 CSV 和 LLM |
| **安全** | 后期建议 | **EncryptedString 值对象**，安全内聚到领域层 |
| **面试话术** | 技术实现导向 | **业务规则 + 架构决策导向**，突出"贫血模型教训"等真实案例 |
